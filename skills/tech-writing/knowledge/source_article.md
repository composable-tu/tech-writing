## Article Outline

### General Rules

1. **Narrative Style**  
   - Should be relaxed, without flirtation, memes, or jokes.  
   - Jokes in educational content distract and annoy.

2. **Verbal Nouns**  
   - Nouns formed from verbs that denote an action, process, or result.  
   - Overload the text, remove action, make it inanimate and dry.  
   - Replace with verbs.

3. **Passive Voice**  
   - Shows action is done *to* the subject.  
   - Makes reading difficult; replace with active voice.

4. **Imperative Mood**  
   - Adds dynamics, stimulates action, addresses reader directly.  
   - Suitable for instructions and manuals.  
   - Allows shorter thoughts.

---

### Structure

1. **Paragraphs**  
   - One thought = one paragraph.  
   - Begin with the main idea.

2. **Short Sentences**  
   - Avoid long sentences.  
   - Text close to spoken language is easier to read.

3. **Syntax**  
   - Simplify syntax.  
   - We pause at punctuation marks.  
   - Read aloud; if you stumble, simplify.

4. **Headings and Lists**  
   - Technical texts should be modular, like code.  
   - Use subheadings and bulleted lists.

5. **Writing the Introduction**  
   - Explain usefulness of the article.  
   - Answer: what concept, what will reader be able to do, why necessary.

6. **Writing the Conclusion**  
   - Draw conclusions and give practical advice.  
   - Provide references to additional materials.  
   - Express personal opinion.

---

### Editing

1. **Accessibility**  
   - Minimize terms and abbreviations.  
   - Explain complex terms, and explain terms used in explanations.  
   - Explain cause and effect.  
   - Avoid the curse of knowledge.

2. **Concision**  
   - Keep introductions short.  
   - Avoid phrases like "pay attention", "it is worth mentioning", "it is important to specify that".

3. **Brevity**  
   - If text can be shortened, shorten it.  
   - Avoid introductory words, participles, de-particles, passive voice, synonyms when describing.

4. **Objectivity**  
   - No biased assessments.  
   - Vague language should be removed or supported by facts.

5. **Clearness**  
   - Be unambiguous and straightforward.

6. **Evaluations**  
   - Refuse subjective evaluations.  
   - Any evaluative judgment needs evidence.

7. **Visualization**  
   - Diagrams, schemes, screenshots, and code samples improve understanding.  
   - Easier to read than text.

---

### Code Examples in Text

1. **Variables**  
   - Avoid `foo` and `bar`.  
   - Use expressive names.  
   - Objects/primitives → noun.  
   - Arrays → plural noun.  
   - Functions → begin with verb.  
   - Event handlers → `on` or `handle` prefix.  
   - Booleans → `is` or `has` prefix.  
   - No magic numbers.

2. **Imports**  
   - Show imports when using code from a module for the first time.

3. **Simplicity**  
   - Start with simple examples, gradually deepen.  
   - Keep code clear and simple.  
   - Comments do not replace text.

4. **Commands**  
   - Always show expected result after terminal commands.

5. **Anti-patterns**  
   - Highlight examples of code not to write.

6. **Relevance**  
   - Regularly update published texts.  
   - Make sure code samples work.

---

### Conclusion

- Technical writing is a skill requiring constant improvement.  
- Make complex things simple; convey thoughts concisely and clearly.  
- Good text is accessible and interesting.  
- Editing and feedback are essential.
