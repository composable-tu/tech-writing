# Distilled Methodology: How to Write Technical Texts

This document distills the writing methodology proposed by the source article, **not** the author's personal writing style.

## Purpose

Provide a repeatable framework for creating technical texts that are clear, concise, accessible, and practical.

## Target Outputs

- Technical articles
- Documentation
- Tutorials
- Blog posts
- Code examples inside texts

## Methodology Overview

```
┌─────────────────────────────────────────────────────────┐
│ 1. GENERAL RULES                                         │
│    - Relaxed narrative, no jokes/memes                   │
│    - Verbs instead of verbal nouns                       │
│    - Active voice instead of passive voice               │
│    - Imperative mood for instructions                    │
├─────────────────────────────────────────────────────────┤
│ 2. STRUCTURE                                             │
│    - One thought per paragraph                           │
│    - Short sentences                                     │
│    - Simplified syntax                                   │
│    - Headings and lists                                  │
│    - Useful introduction                                 │
│    - Actionable conclusion                               │
├─────────────────────────────────────────────────────────┤
│ 3. EDITING                                               │
│    - Accessibility                                       │
│    - Concision                                           │
│    - Brevity                                             │
│    - Objectivity                                         │
│    - Clearness                                           │
│    - Avoid subjective evaluations                        │
│    - Visualization                                       │
├─────────────────────────────────────────────────────────┤
│ 4. CODE EXAMPLES                                         │
│    - Meaningful variable names                           │
│    - Show imports                                        │
│    - Start simple, progress gradually                    │
│    - Show command output                                 │
│    - Include anti-patterns when relevant                 │
│    - Keep examples up to date and working                │
├─────────────────────────────────────────────────────────┤
│ 5. FINAL EDITING                                         │
│    - Read aloud                                          │
│    - Solicit feedback                                    │
│    - Verify all code works                               │
└─────────────────────────────────────────────────────────┘
```

## Application Logic

### When writing from scratch

1. Define the reader and the three introduction questions.
2. Draft the content using active voice, verbs, short sentences.
3. Structure with headings, paragraphs, and lists.
4. Add code examples following the code rules.
5. Edit for accessibility, concision, brevity, objectivity, clearness.
6. Write a conclusion with practical advice and next steps.
7. Read aloud and verify code.

### When reviewing existing text

1. Scan for passive voice and verbal nouns.
2. Check sentence length and syntax complexity.
3. Verify paragraph unity.
4. Check for undefined terms and abbreviations.
5. Remove subjective claims or add evidence.
6. Verify code examples are realistic, runnable, and up to date.
7. Suggest visualizations where text is too dense.

## Key Trade-offs

- **Accessible vs. concise**: Explain terms, but do not over-explain trivial concepts.
- **Complete vs. brief**: Include enough context for code to run, but remove filler.
- **Objective vs. engaging**: Avoid hype, but keep the text interesting through clarity and practical value.
